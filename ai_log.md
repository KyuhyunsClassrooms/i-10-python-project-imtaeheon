## 2차 Gemini 활용

- 사용 날짜:6월 11일
- 사용 목적: 코드 수정, 코드에 궁금한점 질문, 함수와 리스트만들떄 도움받기 등
- 내가 입력한 프롬프트:오늘 저와 함께 프로젝트를 멋지게 완성해 나가는 과정에서 입력하셨던 프롬프트들과, 이에 대해 제가 제안해 드린 핵심 내용을 요약해 드립니다. ai_log.md나 보고서의 AI 활용 기록에 그대로 활용하실 수 있도록 깔끔하게 정리했습니다!

📅 오늘 진행된 AI 활용 기록 요약
1. 코드 문법 오류 및 들여쓰기 디버깅
내가 입력한 프롬프트: (작성하신 while True: 메뉴 및 조건문 코드 입력) "이거 문제가 뭐야"

Gemini 제안 요약: 첫 번째 print문 내 줄 바꿈 문자(\n) 오타 수정, while 반복문 내부 코드의 들여쓰기(Indentation) 누락 문제 지적, 그리고 메뉴 출력이 중복되는 논리적 오류를 바로잡아 정상 작동하는 기본 메인 루프 틀을 제안했습니다.

2. 데이터 타입 예외 처리 (실수 입력 보완)
내가 입력한 프롬프트: "첫째항이나 공차가 정수가 아닐수도있잖아 실수일때도 가능하게 할수있나"

Gemini 제안 요약: 항 번호는 항상 정수(int)여야 하므로 그대로 유지하되, 소수점 단위의 첫째항과 공차/공비도 입력받을 수 있도록 실수 변환 함수인 float(input(...)) 구조를 도입하는 데이터 예외 처리 방안을 제안했습니다.

3. 전체 코드 검증 및 변수 순서 오류 디버깅
내가 입력한 프롬프트: (수열 계산 함수와 메뉴가 통합된 전체 코드 입력) "이거 오류찾아줘", "a 가 정의되지않았대 등비수열에서", "result가 정의되지 않았대"

Gemini 제안 요약: * 등비수열(k == 2) 실행 시 발생한 NameError('a')를 해결하기 위해 사용자 입력 코드가 함수 호출보다 반드시 먼저 실행되어야 하는 실행 순서 규칙을 안내했습니다.

result 변수 에러를 해결하기 위해 각 조건문 울타리(k==1, k==2) 내부에서 계산이 끝나자마자 즉시 history.append()가 일어나도록 코드의 위치를 바로잡아 주었습니다.

이 과정에서 사용된 구조가 수행평가 필수 조건인 2차원 리스트를 완벽히 만족하고 있음을 확인해 주었습니다.

4. 항 번호 입력 예외 처리 설계
내가 입력한 프롬프트: "n,m 에다가 정수가아닌수를 넣으면 어떡해"

Gemini 제안 요약: 사용자가 항 번호에 소수나 문자를 입력했을 때 프로그램이 튕기는 현상을 막기 위해, try-except ValueError 구문을 활용하여 잘못된 타입을 감시하고 예외 메시지를 띄우는 안정성 강화 로직을 제안했습니다.

5. 알고리즘 처리 규칙 요약 (보고서/인터뷰용)
내가 입력한 프롬프트: "입력된 데이터를 어떤 규칙이나 알고리즘으로 처리할 것인가? 이질문에 어떻게 답하지", "간단하게 한문단으로"

Gemini 제안 요약: 조건문(if-elif-else)을 통한 흐름 제어, for 반복문과 if 구간 필터링 조건문을 결합한 구간 합산 알고리즘, 그리고 데이터를 행/열 구조로 안전하게 축적하는 2차원 리스트(history)의 활용 방법을 선생님께 발표하거나 보고서에 적기 좋은 일반 줄글 형태로 요약해 주었습니다.

6. 프로젝트 계획서 및 최종 보고서 명세화
내가 입력한 프롬프트: (계획서 및 보고서 서식 양식 입력) "4.2랑 5번은 글로 쓸수있는 형태로 써줘", "이거 작성해"

Gemini 제안 요약: * 2차원 리스트 history의 각 열 번호(0~5번)가 가지는 수학적 의미와 세 가지 핵심 함수(calculate_arithmetic_range, calculate_geometric_range, show_history)의 명세를 표와 줄글로 작성했습니다.

보고서의 핵심인 '문제 분해 및 모델링', '제어구조의 결합 원리', '실행 결과 분석' 내용을 정교한 텍스트로 채워 양식을 완성해 주었습니다.


- Gemini가 제안한 내용 요약:
- 내가 반영한 부분:2차원리스트, 함수,
- 내가 수정하거나 사용하지 않은 부분:try-except~~ 구문을 사용하라 제안했는데 안배운거라 사용안함
- 반영 위치:main.py에서 코드에.

## 3차 Gemini 활용

사용 날짜: 2026년 6월 16일

사용 목적: 수열 계산기 프로그램의 파이썬 코드 오류(디버깅) 해결, 예외 처리 설계 및 수행평가 최종 결과 보고서 양식 작성 지원

내가 입력한 프롬프트:

Plaintext
(1) 이거 오류찾아줘 / a 가 정의되지않았대 등비수열에서 / result가 정의되지 않았대
(2) n,m 에다가 정수가아닌수를 넣으면 어떡해
(3) 입력된 데이터를 어떤 규칙이나 알고리즘으로 처리할 것인가? 이질문에 어떻게 답하지
(4) 프로그램의 한계에다가 등차또는 등비수열이 아닌수열의 합은 못구한다는 내용으로 써줘
Gemini가 제안한 내용 요약: * 변수 선언 및 함수 호출 순서 오류(NameError)의 원인을 분석하고 올바른 코드 배치 순서를 제안함.

항 번호(n, m)에 정수가 아닌 데이터가 입력되었을 때 프로그램이 다운되는 것을 막기 위해 try-except ValueError 예외 처리 구조를 제안함.

조건문 흐름 제어, 반복문 내 구간 필터링 알고리즘, 2차원 리스트(history) 누적 규칙을 보고서용 줄글 형태로 작성해 줌.

등차·등비 외의 수열(피보나치, 사용자 정의 수열 등)을 계산하지 못하는 현재 프로그램의 수학적 한계점과 이를 극복하기 위한 수식 인식 기능 도입이라는 개선 방안을 제안함.

내가 반영한 부분:

등비수열 구역에서 변수 입력과 함수 호출 순서가 뒤바뀌었던 오류 수정 코드를 그대로 반영함.

항 번호 입력 오류를 잡아내는 try-except 예외 처리 구문 메커니즘을 전면 수용하여 코드에 적용함.

보고서 문항(문제 분해, 2차원 리스트 설명, 함수 설명, 제어구조 활용, 한계와 개선 방안)에 Gemini가 정리해 준 논리적인 설명 문장들을 적극적으로 참고하여 작성함.

내가 수정하거나 사용하지 않은 부분:

Gemini는 처음에 구간 매개변수명을 start_n, end_n으로 제안했으나, 수학 교과서의 정통 표현과 부합하고 직관성을 높이기 위해 내가 직접 n과 m이라는 간결한 변수명으로 수정하여 사용함.

복붙만 하지않고 도움을 요청하며 코드작성, 보고서작성

등비수열 함수 내부에서 다음 항을 준비하는 핵심 연산 수식(current_term *= r) 부분을 Gemini의 가이드라인만 참고한 뒤 내가 직접 추론하여 코드를 완성함.

반영 위치: 파이썬 최종 소스 코드 전반(메인 루프 및 예외 처리 구역), 최종 결과 보고서(report.md)의 각 문항별 답변 구역 전체

## AI 활용 성찰

Gemini가 도움이 된 점: 엔트리같은 블록 코딩을 잘해도 파이썬은 문자열 코딩이다보니깐 모르는 문법같은게 있으면 코딩을 아예 할수가 없다는 단점이 있는데 이 점을
제미나이한테 큰도움을 받았다. 이번 수행평가를 하면서 파이썬에대한 관심도 커지고 ai를 효율적으로 사용하는법도 깨달았다.


Gemini의 제안을 그대로 쓰지 않고 내가 판단하거나 수정한 점: 정보시간에 안배운 내용으로 코드 작성을 추천하길래 그건 안썻고 내가 완전히 이해할수 있는
코드를 작성했다. 그리고 변수이름이 너무 길고 그래서 살짝 줄인 것도 있고 발생할 오류같은걸 물어보면서 꼼꼼히 확인하면서 했다


다음에 AI를 사용할 때 주의해야 할 점: 항상 조심하고있는 부분이지만 ai를 사용하고 막 복붙해서 자신도 모르는 내용을 작성하면
안된다. 내가 이해를하고 도움을 받는 도구로 사용해야하고 대신해주는 걸로 사용하면 안되고 ai의 말이 거짓말인지도 판별하면서 쓰자.
